# ASL Staging Frontend - Internationalization (i18n)

## Outcome
Add or modify translations, work with bilingual (English/Arabic) content, and ensure proper RTL support.

## i18n Architecture
- **Locale routing**: URL-based (`/en/...` and `/ar/...`)
- **Translation system**: JSON dictionary files in `src/i18n/dictionaries/`
- **Backend translations**: WPML plugin on WordPress for product/category content
- **Middleware**: Locale detection and redirect (auto-detects browser language)
- **RTL support**: Automatic based on locale (`ar` = RTL, `en` = LTR)

## Translation Files
- `src/i18n/dictionaries/en.json` - English translations
- `src/i18n/dictionaries/ar.json` - Arabic translations
- `src/i18n/index.ts` - Dictionary loading utility

## Procedure

### 1. Add New Translations
Add the key to BOTH `en.json` and `ar.json`:
```json
// en.json
{
  "section": {
    "newKey": "English text"
  }
}

// ar.json
{
  "section": {
    "newKey": "Arabic text"
  }
}
```

### 2. Use Translations in Components

**Server Components** (pages, layouts):
```typescript
import { getDictionary } from "@/i18n";
const dictionary = await getDictionary(locale);
// Use dictionary.section.newKey
```

**Client Components** (via props):
```typescript
// Pass specific dictionary keys as props from parent server component
interface MyComponentProps {
  dictionary: {
    newKey: string;
  };
}
```

### 3. RTL Styling
Use Tailwind's `rtl:` variant or logical CSS properties:
```tsx
// Tailwind RTL variant
<div className="ml-4 rtl:ml-0 rtl:mr-4">...</div>

// Better: Use logical properties
<div className="ms-4">...</div>  // margin-inline-start (left in LTR, right in RTL)
<div className="me-4">...</div>  // margin-inline-end
<div className="ps-4">...</div>  // padding-inline-start
<div className="pe-4">...</div>  // padding-inline-end
```

### 4. Layout Direction
The locale layout (`src/app/[locale]/layout.tsx`) sets `dir` and `lang`:
```tsx
<div dir={dir} lang={validLocale}>
```

### 5. WPML Backend Content
- Product names, descriptions, categories are translated via WPML in WordPress
- Pass `locale` parameter to all API calls for localized content
- Category slugs differ between locales (see slug mappings in `src/lib/api/woocommerce.ts`)
- WPML uses different product/category IDs per locale - always use locale-aware API calls

## Specifications
- Default locale: `en`
- Supported locales: `en`, `ar` (defined in `src/config/site.ts`)
- Arabic text direction: RTL (right-to-left)
- English text direction: LTR (left-to-right)
- WPML language parameter: `?lang=en` or `?lang=ar` on API requests
- Category slug mappings (English <-> Arabic) are hardcoded in `src/lib/api/woocommerce.ts`

## Advice
- Always add translations to BOTH locale files simultaneously
- Test UI in both locales after any layout change
- Arabic text may be longer or shorter than English - test for overflow
- Numbers, dates, and currencies should be formatted locale-appropriately
- Use logical CSS properties (`ms-`, `me-`, `ps-`, `pe-`) over physical ones (`ml-`, `mr-`, `pl-`, `pr-`) for automatic RTL support
- The `localeConfig` in `src/config/site.ts` provides `name`, `dir`, and `hrefLang` for each locale
- If adding a new category, update the slug mapping in `src/lib/api/woocommerce.ts`

## Forbidden Actions
- Do not hardcode text strings in components - always use the dictionary system
- Do not assume LTR layout - always test in both directions
- Do not use physical CSS properties (ml/mr/pl/pr) when logical properties (ms/me/ps/pe) would work
- Do not forget to add translations to both `en.json` and `ar.json`
